# 分類與平台規則

Novae 的提案與設備報修分類是安裝後的動態資料，不是 repository 內的硬編碼或 JSON。首次部署完成後，由 `ADMIN_EMAILS` 對應的首位平台管理員登入並完成引導設定；之後在「我的 → 系統設定」維護。

## 首次設定

1. 完成後端與前端部署。
2. 使用 `ADMIN_EMAILS` 中的帳號首次登入。
3. 步驟 1：確認介面語言；系統會依瀏覽器／作業系統首選語言預選，也可在繼續前切換。
4. 在提案／設備切換介面中選擇要啟用的平台功能。兩者預設啟用；也可只啟用其中一項，或兩者都關閉而只使用公告。
5. 使用與系統設定相同的分類清單與編輯表單，為每個已啟用功能建立至少一個分類，填永久且唯一的 ID、顯示名稱並指定唯一預設分類。
6. 提案分類另選閱讀範圍、作者顯示、留言、附議門檻／天數與回覆天數。
7. 按下完成時會先提示負責人指派將略過；等需要負責的人都已註冊並首次登入後，再到「系統設定 → 人員與管理權限」逐分類新增。

未啟用的功能不要求填寫分類；之後可在「系統設定 → 分類與處理流程」的平台功能區重新開啟。管理員指派刻意不放在首次設定的必要步驟，因為其他管理員可能尚未登入註冊。完成請求可安全重試；若資料其實已寫入但回應中斷，重新整理或再按一次會讀回已完成狀態，不會建立第二套設定。

## 平台功能開關

| 功能 | 關閉後的行為 |
| --- | --- |
| 提案 | 側欄與手機導覽隱藏提案入口；既有提案、分類與權限保留 |
| 設備報修 | 側欄與手機導覽隱藏設備入口；既有報修、分類與權限保留 |

公告不受這兩個開關影響。功能開關與分類草稿會在系統設定一次儲存，不會只寫入其中一部分。

## 規則何時能修改

| 規則 | 建立後 | 影響範圍 |
| --- | --- | --- |
| 分類 ID | 鎖定 | 保護既有網址、關聯與通知 |
| 閱讀範圍 | 永久鎖定 | 避免既有內容因改設定意外曝光 |
| 作者顯示 | 永久鎖定 | 避免匿名承諾被日後改寫 |
| 留言開關 | 可修改 | 只影響之後建立的提案；舊留言保留 |
| 附議與期限 | 可修改 | 只影響之後建立的提案 |
| 名稱 | 可修改 | 立即更新分類顯示 |
| 平台功能開關 | 可修改 | 只影響導覽與新入口；既有資料保留 |

提案建立時會把留言、附議、期限及隱私規則快照到案件本身。因此關閉某分類的留言不會刪除舊留言，也不會關閉既有提案原本允許的留言。分類管理員仍可在單筆提案中關閉或重新開放「新留言」；關閉後既有留言保持可讀。

## 閱讀範圍

- `school`：已登入且符合校內網域的使用者可讀。
- `reviewed-school`：審核前僅作者與管理員可讀，通過後才進入校內列表。
- `owner-admin`：內容只供作者與該分類管理員處理；作者身分固定對管理端可見。

這些值由後端與資料庫執行，不依賴前端隱藏按鈕。

## 分類管理員與通知

「系統設定 → 人員與管理權限」先選提案或設備分類，再顯示目前已取得權限的所有使用者。可在同一處搜尋已註冊使用者並新增、修改通知設定或快速撤銷；同一人可負責多個分類，同一分類也可有多人負責。查找時可使用姓名、Email 或 UID。

提案分類負責人會收到該分類的新提案通知；設備分類負責人只有在該指派開啟「新案件通知」時收到新報修通知。兩者都排除案件作者。平台總管理員仍可處理所有分類，但不會只因總管理員身分收到新提案或新報修通知；若需要收件，也要在該分類明確指派自己。平台總管理員名單只能改部署環境的 `ADMIN_EMAILS`，程式內沒有授予或撤銷入口。

使用校園信箱或 UID 找不到人時，請對方先登入一次，讓系統建立使用者資料後再指派。

## 升級既有安裝

migration 會自動把既有提案分類、分類指派及設備案件轉成動態資料，並補上平台功能開關的預設狀態。既有提案會依原規則補上隱私、留言、附議及期限快照；設備案件會歸入相容的預設分類。不要手動重寫已部署 migration。

分類沒有封存或停止接案狀態。需要停止整個提案或設備入口時使用平台功能開關；刪除分類則是不可復原的永久刪除，會一併移除該分類內的案件、留言、附議／受影響者關聯、通知與圖片資料，預設分類必須先轉移後才能刪除。

## 限流與圖片

`config/rate-limits.config.json` 仍控制操作限流、圖片數量與瀏覽器壓縮；它不是分類設定。修改後依 repository 的 codegen 與驗證流程提交。Cloudflare burst limits 仍位於 `cloudflare/wrangler.toml`。

## 驗收

1. 為公開、審核後公開與私密提案分類各建立一件測試提案。
2. 確認一般使用者、作者與分類管理員看到的內容符合規則。
3. 確認留言與附議只依案件建立時的設定運作，並確認單筆提案關閉新留言後仍能查看舊留言。
4. 為每個提案與設備分類建立測試案件，確認只有明確指派且符合通知設定的負責人收到建立通知；未指派的平台總管理員不應收到。
5. 確認設備列表可按分類切換，從目前分類建立的報修會保留該分類，且正確負責人能留言、更新狀態與刪除。
6. 建立專用測試分類與案件後刪除分類，確認案件及其留言、附議／受影響者、通知與圖片資料都已移除，且預設分類不能直接刪除。
7. 關閉再開啟提案或設備功能，確認導覽隱藏／恢復，且既有案件與分類仍可管理。
