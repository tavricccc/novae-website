# 分類與平台規則

Novae 的提案與設備報修分類是安裝後的動態資料，不是 repository 內的硬編碼或 JSON。首次部署完成後，由 `ADMIN_EMAILS` 對應的首位平台管理員登入並完成引導設定；之後在「我的 → 平台管理中心」維護。

## 首次設定

1. 完成後端與前端部署。
2. 使用 `ADMIN_EMAILS` 中的帳號首次登入。
3. 建立至少一個提案分類與一個設備報修分類。
4. 每個分類填永久且唯一的 ID、顯示名稱與說明。
5. 提案分類另選閱讀範圍、作者顯示、留言、附議門檻／天數與回覆天數。
6. 完成設定後到「平台管理中心 → 人員與管理權限」，用已註冊使用者的校園信箱或 UID 指派分類管理員。

管理員指派刻意不放在首次設定的必要步驟，因為其他管理員可能尚未登入註冊。

## 規則何時能修改

| 規則 | 建立後 | 影響範圍 |
| --- | --- | --- |
| 分類 ID | 鎖定 | 保護既有網址、關聯與通知 |
| 閱讀範圍 | 永久鎖定 | 避免既有內容因改設定意外曝光 |
| 作者顯示 | 永久鎖定 | 避免匿名承諾被日後改寫 |
| 留言開關 | 可修改 | 只影響之後建立的提案；舊留言保留 |
| 附議與期限 | 可修改 | 只影響之後建立的提案 |
| 名稱與說明 | 可修改 | 立即更新分類顯示 |
| 使用中／封存 | 可修改 | 封存後不接受新案件，舊資料保留 |

提案建立時會把留言、附議、期限及隱私規則快照到案件本身。因此關閉某分類的留言不會刪除舊留言，也不會關閉既有提案原本允許的留言。分類管理員仍可在單筆提案中關閉或重新開放「新留言」；關閉後既有留言保持可讀。

## 閱讀範圍

- `school`：已登入且符合校內網域的使用者可讀。
- `reviewed-school`：審核前僅作者與管理員可讀，通過後才進入校內列表。
- `owner-admin`：內容只供作者與該分類管理員處理；作者身分固定對管理端可見。

這些值由後端與資料庫執行，不依賴前端隱藏按鈕。

## 分類管理員與通知

「平台管理中心 → 人員與管理權限」可分別勾選某位使用者負責的提案分類與設備報修分類。設備分類管理員會收到該分類的新報修通知，並可回覆、更新狀態或刪除該分類案件。平台管理員仍擁有全部分類權限。

使用校園信箱或 UID 找不到人時，請對方先登入一次，讓系統建立使用者資料後再指派。

## 升級既有安裝

migration 會自動把既有提案分類、分類指派及設備案件轉成動態資料。既有提案會依原規則補上隱私、留言、附議及期限快照；設備案件會歸入相容的預設分類。不要手動重寫已部署 migration。

## 限流與圖片

`config/rate-limits.config.json` 仍控制操作限流、圖片數量與瀏覽器壓縮；它不是分類設定。修改後依 repository 的 codegen 與驗證流程提交。Cloudflare burst limits 仍位於 `cloudflare/wrangler.toml`。

## 驗收

1. 為公開、審核後公開與私密提案分類各建立一件測試提案。
2. 確認一般使用者、作者與分類管理員看到的內容符合規則。
3. 確認留言與附議只依案件建立時的設定運作，並確認單筆提案關閉新留言後仍能查看舊留言。
4. 為每個設備分類建立測試報修，確認正確管理員收到通知且能更新狀態。
5. 封存測試分類，確認舊案件仍可讀、新案件不能再選該分類。
