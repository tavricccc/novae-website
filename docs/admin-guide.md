# 管理員流程

平台總管理員身分只來自後端 `ADMIN_EMAILS`，不是前端按鈕，也不能在程式內授予或撤銷。更新環境變數後，使用者必須重新登入，才能取得更新後的角色。

## 1. 每天先看三個地方

1. 通知：是否有自己被明確指派分類的待處理事件；平台總管理員不會自動收到所有新案件。
2. 提案：先篩選「待審核」與「未回覆」。
3. Dashboard：背景工作、通知、圖片或外部同步是否異常。

## 2. 審核提案

1. 打開待審核提案。
2. 確認分類、隱私、個資、內容與圖片是否適合公開。
3. 使用「審核提案」操作。
4. 通過後，審核型分類會向校內開放；若有附議，會開始分類設定的附議期限。
5. 不通過時填寫清楚原因；提案成為「審核未過」，只有作者與管理員可繼續查看。

## 3. 更新狀態與回覆

1. 打開提案詳情。
2. 將「未回覆」更新為「處理中」，並提供可理解的進度。
3. 完成時選「已完成」並寫可驗證結果。
4. 無法執行時選「無法實行」，說明限制、評估依據與可能替代方案。

有設定 `responseDeadlineDays` 的分類，期限會從建立或附議達標開始，由分類是否附議自動決定。

需要暫停討論時，可在單筆提案的管理操作中關閉新留言；既有留言不會被刪除，之後也能由該分類管理員重新開放。

## 4. 發布公告

1. 選全域「新增」。
2. 管理員會看到「公告」選項。
3. 填標題、內容與圖片後發布。
4. 公告發布後不可直接編輯；若內容錯誤，確認影響後刪除並重新發布。

公告頁與通知頁是兩個獨立頁面。刪除使用紅色文字與圖示的警告操作，確認後才執行。

## 5. 刪除內容

1. 先確認刪除對象、作者、附件與外部副本。
2. 從提案或公告的管理選單進入刪除。
3. 確認 dialog 會清楚標示警告。
4. 刪除後，資料庫與 Cloudinary 清理由受控流程與 deletion worker 完成，不要手動只刪一邊。

## 6. Dashboard 怎麼看

依序確認：

- 主要資料讀寫與錯誤。
- outbox 是否累積，以及最舊事件等待多久。
- FCM、Notion、Cloudinary 等外部副作用是否失敗。
- 排程維護與刪除工作是否正常。
- 使用量是否接近服務額度。

## 7. 分類負責人

在「平台管理中心 → 人員與管理權限」先選提案或設備分類。畫面會列出目前所有負責人，可搜尋已登入過的校內使用者來新增，也可直接修改設備新案件通知或撤銷權限。同一分類應依實際交接需要配置多人；分類權限不會讓使用者變成平台總管理員。

新提案只通知該提案分類的明確負責人；新設備回報只通知該設備分類中開啟通知的負責人。作者會排除，未被指派的平台總管理員也不收件。完整事件矩陣見[使用者流程](user-guide.md#每種通知會發給誰)。

## 8. 交接

- 維持至少兩位受信任管理員，避免單點故障。
- 離職或職務變動時更新 `ADMIN_EMAILS`、GitHub Environment 權限與各服務 token。
- 不共用個人密碼；用服務帳號、整合 token 與最小權限。
- 每學期重新確認分類、附議門檻、天數、回應責任與隱私告知。

日常與事故處理請接著讀[上線後維運](operations.md)。
