# 使用者流程

這一頁依 App 內的實際順序說明。桌面使用側邊欄；手機使用底部浮動導覽，中央「新增」只有加號、沒有文字。公告、通知、設定都是各自獨立頁面。

## 1. 登入

1. 打開正式網址。
2. 選擇 Google 登入；按鈕會進入「登入中…」並暫時反灰。
3. 使用符合學校允許網域的帳號。
4. 若瀏覽器改走 redirect 流程，從 Google 返回後按鈕仍會維持 loading，直到角色與分類目錄同步完成。
5. 同步完成後會自動離開登入頁並進入預設提案分類；手機底欄與側欄也在這之後才出現。

不符合網域的帳號會被拒絕。若管理員剛加入 `ADMIN_EMAILS`，需要重新登入取得新角色。

首位平台總管理員第一次進入尚未設定的平台時，會先確認語言，再選擇是否啟用提案與設備報修，最後只為已啟用的功能建立分類。預設語言取瀏覽器／作業系統的第一個首選語言；完成設定若遇到網路中斷，可重新整理或重試，系統會辨識已完成狀態。

## 2. 瀏覽提案

1. 從提案頁選擇分類，或進入「我的提案」。
2. 使用搜尋、狀態篩選與排序縮小結果。
3. 桌面版以獨立卡片排列；手機版保留同一份狀態、作者、日期、附議進度與操作資訊。
4. 打開卡片查看詳情；內容與討論在手機以可展開的圖示文字控制切換。

看不到某件提案不一定是故障：待審核或私密案件只對作者與管理員可見。

## 3. 建立提案

1. 選擇全域「新增」。
2. 選「提案」，再選分類。
3. 填標題與內容；標題最多 30 個字、內文最多 1000 個可見字元，需要時可加入圖片。
4. 確認分類與內容後發布。
5. 按鈕會在原位顯示「發布中」，完成或失敗會在底部回饋列說明。

分類會改變送出後的流程：

| 設定 | 實際結果 |
| --- | --- |
| `school` | 校內登入者可以依列表與狀態規則閱讀 |
| `reviewed-school` | 先顯示待審核；通過後才校內可讀 |
| `owner-admin` | 只在作者自己的提案與管理端可見 |
| `authorVisible: false` | 一般讀者看不到作者；管理端仍保留關聯 |
| `support.enabled: true` | 顯示分類設定的門檻人數與剩餘天數 |

## 4. 附議與留言

- 只有開啟附議的分類會顯示附議進度與按鈕；使用者可以附議或取消。
- 達到 `support.goal` 就提前成功；超過 `support.deadlineDays` 未達標會成為「未通過」。
- 審核型分類在核准前不向全校開放留言；私密分類的留言只在作者與管理員之間可見。
- 留言可回覆，採純文字輸入、不解析 Markdown，也不需要切換預覽；每則最多 70 個可見字元。
- 留言仍可附加圖片；目前每則最多 1 張，實際張數由 `rate-limits.config.json` 控制。
- 手機版的留言數直接顯示在內容／留言 segmented control，不會在留言面板內重複顯示。

## 5. 看懂狀態

| 狀態 | 意義 |
| --- | --- |
| 待審核 | 管理員尚未決定是否公開 |
| 審核未過 | 管理員拒絕公開；作者可查看原因 |
| 未回覆 | 已進入應處理階段，但尚未提供處理回覆 |
| 處理中 | 管理單位已開始處理 |
| 未通過 | 附議期限已到但未達門檻 |
| 已完成 | 已結案並提供結果 |
| 無法實行 | 已結案並說明限制或替代方向 |

## 6. 設備回報

設備看板和提案看板一樣先選分類，分類會保留在網址與新增流程中。從目前分類新增報修時可填內容與圖片；詳情頁可留言、標記「我也遇到」並追蹤狀態。只有該設備分類的負責人或平台總管理員能管理案件，負責人的新案件通知可依分類指派開關。

## 7. 公告

公告有獨立列表與詳情頁。使用者可以閱讀、分享、按讚與留言；只有管理員能從全域新增選單發布公告或從管理選單刪除。新增公告同樣限制標題 30 個字、內文 1000 個可見字元。

## 8. 通知與我的

- 通知頁用一張大卡片分隔每則通知；打開頁面會同步已讀狀態，沒有每則「標記已讀」按鈕，也不在列表顯示時間。
- 允許瀏覽器通知後可接收 Web Push；拒絕也不影響站內通知。
- 「我的」頁可調整通知與 App 設定。語言以 dropdown 顯示目前語系與所有可用選項，新語系可直接加入同一清單。登出是單一獨立按鈕，不另包第二層卡片。

### 每種通知會發給誰

| 事件 | 收件人 |
| --- | --- |
| 新公告 | 所有已登入、可接收廣播通知的校內使用者 |
| 公告留言 | 回覆時通知上層留言作者；否則通知公告作者；都排除操作者 |
| 新提案 | 該提案分類明確指派的負責人，排除提案作者；平台總管理員不會自動收到 |
| 提案留言 | 回覆時通知上層留言作者；否則通知提案作者；都排除操作者 |
| 提案狀態變更 | 提案作者與目前附議者，排除操作者 |
| 附議達標 | 提案作者與所有附議者 |
| 提案刪除 | 提案作者與刪除前的附議者，排除操作者 |
| 新設備回報 | 該設備分類中開啟新案件通知的負責人，排除作者；平台總管理員不會自動收到 |
| 設備狀態變更 | 回報作者與所有標記「我也遇到」的使用者 |

站內通知與 Web Push 使用同一收件邏輯；關閉裝置通知只停止 Push，不會改變站內通知的授權與保存。

遇到問題時先記錄「哪個角色、哪個分類、哪個狀態、哪個動作」，再交給管理員依[排錯流程](troubleshooting.md)檢查。
