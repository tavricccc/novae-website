# 專案總覽與亮點

學生權益提案平台是一個給校內社群使用的開源提案、附議、公告與維運平台。它的目標不是只做一個表單，而是讓校內公共議題可以被分類、追蹤、回覆與備份，同時保護私密案件與作者資訊。

## 解決的問題

傳統學生提案流程常見問題包括：

- 提案散落在社群貼文、表單或私訊，後續追蹤困難。
- 公共議題、設備需求、權益維護案件混在一起，處理規則不清楚。
- 作者隱私與公開討論之間難以平衡。
- 管理員需要手動整理統計、通知、備份與同步紀錄。
- 系統部署後缺少維運視角，錯誤常要回頭翻雲端後台。

本專案把這些流程整理成可設定、可審核、可追蹤的 Web App，並提供管理員 Dashboard 觀察平台狀態。

## 核心功能

| 面向 | 功能 |
| --- | --- |
| 提案 | 新增提案、分類瀏覽、搜尋、留言、分享、我的提案、狀態追蹤 |
| 附議 | 依分類設定啟用附議、設定門檻與期限、支援取消附議與進度顯示 |
| 審核 | 公共議題可先審核再公開，管理員可調整狀態與回覆期限 |
| 隱私 | 支援作者匿名顯示、私密作者資料分離保存、只讓本人與管理員讀取敏感內容 |
| 公告 | 管理員發布公告，使用者可查看、留言與按讚 |
| 通知 | App 內通知與 Web Push，依事件通知作者、管理員或所有使用者 |
| 圖片 | 前端壓縮、Cloudinary authenticated upload、後端簽名 URL 顯示 |
| 備份 | Notion 同步提案、公告與狀態，刪除時保留紀錄並標記 |
| 維運 | Dashboard 顯示統計、同步狀態、近期異常、維護排程與分類成果 |

## 專案亮點

- **Config 驅動流程**：分類、讀取權限、作者顯示、附議門檻、期限與限流都集中在 `config/`，不需要到處改程式碼。
- **前後端安全邊界清楚**：前端只負責 UI 與公開設定，寫入與敏感讀取都經過 Supabase Edge Functions。
- **資料權限不依賴前端判斷**：前端顯示條件只是體驗，真正權限由 Firebase token、後端 action 與 Supabase RLS 保護。
- **背景工作可追蹤**：通知、推播、Notion 同步、圖片清理與維護工作透過 outbox 與維護紀錄追蹤。
- **部署自動化**：GitHub Actions 分開部署前端與後端，正式環境使用 `production` Environment 管理 secrets。
- **適合低成本啟動**：多數服務可從免費或低用量方案開始，流量成長後再逐步升級。
- **面向行動裝置**：PWA、Service Worker、Web Push、手機底部導覽與全螢幕詳情對話框支援。
- **可接手維護**：`structure.md`、架構測試、型別檢查、lint、build 與 Edge Function check 提供接手線索。

## 適合使用的情境

- 學生會、系學會、自治組織需要公開提案與附議平台。
- 校內單位需要分類處理公共議題、設備問題與權益案件。
- 團隊希望用低成本雲端服務部署可維護的校內平台。
- 開源社群想參考 Vue + Supabase + Firebase Auth 的完整實作。

## 不適合的情境

- 需要完全離線或校內自架、不能使用外部雲端服務。
- 需要法規等級的正式申訴系統、電子簽章或完整公文流程。
- 需要多租戶 SaaS 計費、跨校組織管理或企業級審計報表。
- 不能接受 Firebase、Supabase、Cloudinary、Notion、Vercel、Upstash 等外部服務依賴。

## 成熟度設計

這個專案刻意把「能上線維運」需要的事項納入設計：

- 有正式部署文件與 secret 來源說明。
- 有前端、後端與架構測試指令。
- 有 GitHub Actions 自動部署。
- 有管理員 Dashboard 觀察近期異常與維護狀態。
- 有資料保留與清理策略。
- 有設定集中化與產生型別的流程。
- 有風險較高的 reset workflow 警示。
